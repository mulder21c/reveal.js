What is WAI-ARIA

우선 WAI-ARIA가 무엇인가? 에 대해서 먼저 가볍게 알아보고 가려고 합니다.
WAI
Web Accessibility Initiative
ARIA
Accessible Rich Internet Application
WAI-ARIA는 이 약어이구요, W3C의 WAI에서 제정한 RIA의 접근성 보장을 위한 기술 규격 입니다.
WAI-ARIA는 2006년에 First Working Draft가 시작되었구요, 2014년 3월에 1.0 버전이 recommendation으로 등록되었습니다.

현재 1.1 버전이 Proposal Recommendation 버전으로 등록되어 있고, W3C 문서는 최신 권고안을 볼 수 있는 대표 주소가 있는데, 이 주소로 들어가보면 보통은 Recommendataion 버전의 문서가 보여지는데 WAI-ARIA는 이 PR 버전으로 접근이 됩니다. 즉, 현재는 1.1이 최종 권고안이라고 해도 될 것 같습니다.

WAI-ARIA 가 등장한지 거의 10년이 되었는데 WAI-ARIA 발표를 해보면 WAI-ARIA를 처음 들어본다 하시는 분들이 많은 것 같더라구요.

Why use?

그럼 이걸 왜 사용하는가?

이를 알아야 이것이 어디에 쓰이는 물건이고 사용하면 뭐가 좋기에 이걸 앞에 나와서 설명하고 있는가를 알 수 있겠죠?

supplement for native language semantics, not a replacement

W3C의 WAI-ARIA 명세에서 WAI-ARIA는 네이티브 언어의 의미론을 보완하기 위한 것이라 기술하고 있습니다.

그리고 추가로 교체가 아니라고 기술되어 있는데요, 이것은 WAI-ARIA가 native language의 의미를 교체하는데 사용하는 것이 아니라는 이 점을 이야기 하구요, 이를 잘 기억하고 있어야 WAI-AIRA를 오용하는 일을 줄일 수 있습니다. 가능은 하나 의도와는 맞지 않다라는 것입니다.

Role, Property, State

WAI-ARIA는 Role, Property, State이 세 가지를 통해 정보를 전달하게 됩니다.

Role

Attaching a role gives assistive technologies information about how to handle each element

Role이란 것은 우리말로 하면 역할이죠.

WAI-ARIA에서 Role을 부여하는 것은
각 요소가 어떤 역할이다 라는 것을 부여하여
스크린리더 같은 보조 기술에 각 요소를 어떻게 처리해야 한다는 정보를 제공합니다.

예를 들어, button role을 주었다면 이 요소에는 button 역할을 부여하고 버튼으로서 처리를 해야 한다는 정보를 제공하는 것이죠.

<tag role="keyword">
<div role="navigation">
list of role keywords

role을 적용하는 것은 단순히 attribute를 적용하듯 사용합니다.

(클릭)

예를 들어 div 요소에 navigation role을 부여하기 위해 이런 식으로 사용할 수 있습니다.

이 역할은 아무거나 임의로 적용하는 것은 아니구요, 정해져 있는 키워드들이 있습니다. (클릭)

Property, State

Both provide specific information about an object, and both form part of the definition of the nature of roles.

Property와 State 즉, 속성과 상태는 "객체에 대한 특정한 정보를 전달하고, 역할의 특징에 대한 정의의 일부를 형성합니다." 라고 W3C에 나와 있는데요
쉽게 말해, 해당 객체에 대한 속성(성질)이나 현재 상태를 부여합니다.

예를 들어, 객체의 레이블을 정의하거나, 하위 메뉴를 가지고 있는지의 속성을 설정하거나, 현재 펼쳐진 상태인지 닫혀진 상태인지 등을 설정합니다.

<tag aria-*="value">
<button aria-expanded="false" aria-controls="sect1">

property와 state는 aria- prefix를 사용하여 attribute 이름으로 사용합니다.

(클릭)

button 요소에 펼침 상태를 접힘 상태 즉 false로 부여하고, 이 버튼이 sect1 이라는 ID를 가진 요소에 대한 컨트롤이라는 속성을 부여하기 위해 이렇게 사용합니다.

How to improve a11y with WAI-ARIA

그럼 이제, 이 WAI-ARIA를 가지고 접근성을 향상시키기 위해 어떻게 해야 하는지를 보도록 하겠습니다.

Step 1

use native HTML

첫 번째는 일단 native HTML을 사용하는 겁니다.
정말 불가능한 경우가 아닌 이상 native HTML을 사용하여 semantic markup을 하는 것이 시작이라 할 수 있습니다.

앞서 WAI-ARIA를 왜 사용해야 하는가를 이야기 할 때,
네이티브 언어의 의미론을 보완하기 위한 것이라고 말씀을 드렸는데요, WAI-ARIA의 목적이 보완이기 때문에 native HTML을 사용해야하는 것이 가장 기본이됩니다.

then, when use WAI-ARIA?

  • the feature is available in HTML but it is not implemented or it is implemented, but accessibility support is not.
  • visual design constraints rule out the use of a particular ative element
  • feature is not currently available in HTML

그렇다면, 언제 WAI-ARIA를 사용하게 될까요?

  • (클릭) 첫째로, 기능이 HTML로 가능하지만 아직 구현되지 않았거나, 구현은 되어 있지만 브라우저에서 접근성 지원이 되지 않은 경우
  • (클릭) 둘째로, 커스텀 컴포넌트와 같이 시각적 디자인의 이유로 특정 네이티브 요소를 사용할 수 없는 경우
  • (클릭) 셋째로, 기능이 현재 HTML로 사용할 수 없는 경우

예를 들어, 탭 콘텐츠 같은 것은 HTML 명세로는 이것이 탭 기능이다라는 것을 전달할 수가 없죠. 이런 경우 WAI-ARIA를 사용하여 해당 정보를 보완해 줄 수 있게 됩니다.

Step 2

Add ARIA

native HTML를 사용한 후, 이것만으로 제공되기 어려운 정보들을 ARIA를 추가하여 더 향상된 접근성을 제공합니다.

inline or via Script?

  • If the ARIA role or aria-* attribute does not rely on scripting to provide interaction behavior, then inline.
  • If the content and interaction is only supported in a scripting-enabled browsing context, then inline.
  • If the ARIA role or aria-* attribute can be inserted, changed and removed ARIA via scripting, then via script.

그런데 앞서 언급한 바와 같이 ARIA는 동적인 웹 페이지에 관련된 것이기 때문에 inline으로 미리 markup에 두어야 하는가? 혹은 script에 의해 DOM에 추가해야 하는가라는 궁금증이 생기는 데요, W3C에서는 크게 다음의 두 가지 기준을 제공하고 있습니다.

(클릭) ARIA role이나 aria-* 속성이 인터랙션을 제공하는데 스크립트에 의존하지 않는다면 inline으로,

(클릭) 콘텐트와 인터랙션이 스크립트 활성화 브라우징 콘텍트스에서만 지원되는 경우에도, inline으로 하는 것이 안전하고,

(클릭) ARIA role이나 aria-* 속성이 스크립트를 통해 추가,변경,제거 되어야 하는 경우에는 스크립트를 통해서

작성하도록 권장하고 있습니다.

Step 3

developing keyboard interface

그리고 중요한 것이 바로 키보드 인터페이스를 개발해야 합니다.

requirements

  • All interactive ARIA controls must be usable with the keyboard.
  • If can click or tap or drag or drop or slide or scroll, must also be able to perform an equivalent action using the keyboard
  • All interactive widgets must be scripted to respond to standard key strokes or key stroke combinations where applicable
  • 모든 대화형 aria 컨트롤들은 키보드로 사용가능해야 하고
  • 클릭 할 수 있거나 탭, 드래그, 드롭, 슬라이드, 스크롤 할 수 있다면, 키보드를 사용해서 동등한 수준의 행위를 할 수 있어야 합니다.
  • 대화형 위젯은 표준 키 스트록이나 키 스트록 조합에 응답하도록 스크립팅 되어야 합니다.

예를 들어, button role를 부여한 요소가 있다면 이는 space나 enter키로 활성화가 가능해야 합니다.

Use ARIA Technique - developing keyboard interface

Keyboard Support
Key Function
Tab
  • when focus moves into the tab list, place focus on active tab element
  • When the tab list contains the focus, moves focus to the next element in the tab sequence, which is the tabpanel element
Right Arrow
  • Moves focus to the next tab.
  • If focus is on the last tab, moves focus to the first tab.
  • Activates the newly focused tab
Left Arrow
  • Moves focus to the previous tab.
  • If focus is on the first tab, moves focus to the last tab.
  • Activates the newly focused tab
Home Moves focus to the first tab and activates it
End Moves focus to the last tab and activates it
